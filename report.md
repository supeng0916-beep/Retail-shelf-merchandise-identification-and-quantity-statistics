# 零售货架商品识别与数量统计项目汇报

## 1. 项目概述

本项目面向零售货架与冰柜场景，目标是实现一套可落地的商品识别与数量统计系统。系统接收一张真实货架图片后，能够自动完成商品目标检测、SKU 匹配、数量统计以及可视化展示，为门店巡检、陈列核查和库存盘点提供自动化支持。

当前项目已经完成从前端上传、后端识别、SKU 库管理到结果展示的完整闭环，核心技术路线为：

- `YOLO`：负责从货架图中检测商品区域
- `CLIP`：负责提取商品视觉特征
- `FAISS`：负责在 SKU 特征库中做快速检索
- `OCR`：负责读取包装上的文字，辅助区分相似商品
- `FastAPI + Web Demo`：负责识别服务与交互展示

本报告基于当前的 **Full Library Mode（全库模式）** 编写，即使用主配置、主 SKU 图片库和主索引进行评估，不包含此前为专项演示准备的聚焦模式结果。

## 2. 业务目标

项目要解决的核心问题是：在复杂零售场景中，如何以较低人工成本识别货架商品、定位具体 SKU，并输出数量统计结果。

对应的业务价值包括：

- 降低人工盘点和货架巡检成本
- 提高商品识别与陈列核查效率
- 为后续缺货检测、陈列分析、门店执行监控提供基础能力
- 支持从单品验证逐步扩展到更大 SKU 库的识别系统

## 3. 当前项目范围

目前系统已经具备以下能力：

- 对货架图中的商品进行目标检测
- 基于 SKU 图片库进行 SKU 级别匹配
- 对同类商品进行数量统计
- 对不满足匹配条件的商品输出 `Unknown`
- 通过商品管理页进行 SKU 图片增量导入
- 基于前端页面完成上传、识别和结果查看

当前全库模式的基础数据规模如下：

- SKU 元数据：`386` 个
- SKU 图片目录：`386` 个
- SKU 参考图片：`1514` 张

主库当前采用以下资源运行：

- 主配置：`config.yaml`
- 主索引：`index/faiss_clip.index`
- 主运行库：`sku_runtime.db`
- 主 SKU 图库：`database/sku_images`

## 4. 系统架构

### 4.1 整体架构

系统采用“两阶段识别”架构：

1. 第一阶段由 `YOLO` 检测货架图中的商品候选框  
2. 第二阶段对每个候选框提取 `CLIP` 特征，并在 `FAISS` 向量库中检索最接近的 SKU  
3. 当视觉匹配分数处于不确定区间时，再调用 `OCR` 读取包装文字并进行加权修正  
4. 最终输出商品名称、SKU 识别结果、数量统计和标注图

### 4.2 核心模块分工

- `src/detector.py`
  - 封装 YOLO 检测模型
  - 输出货架图中的商品框位置与置信度
- `src/embedder.py`
  - 使用 CLIP 提取商品裁剪图的 512 维特征
- `src/matcher.py`
  - 使用 FAISS 做 Top-K 检索与投票决策
  - 支持 `Unknown` 拒识
- `src/ocr_booster.py`
  - 对包装文字进行识别并辅助修正视觉匹配
- `api/routers/detect.py`
  - 串联检测、匹配、OCR、计数与结果返回
- `demo/`
  - 提供识别中心与商品管理页面，支持可视化演示

## 5. 核心识别流程

一次完整识别流程如下：

1. 用户上传货架图片
2. 系统预处理图片并限制最大尺寸
3. YOLO 输出商品检测框
4. 对每个检测框裁剪图像并批量送入 CLIP
5. FAISS 返回候选 SKU，并由 Top-K 投票机制给出初步结果
6. 当 CLIP 分数落入设定区间时，触发 OCR 辅助判断
7. 系统根据阈值、分数差和投票结果决定是否输出具体 SKU 或 `Unknown`
8. 前端展示标注图、商品名称和数量统计

## 6. Full Library Mode 当前表现

### 6.1 当前状态

目前主索引已经基于全库完成重建，全库模式能够正常运行完整链路：

- 货架图上传与检测正常
- 主索引检索正常
- `Unknown` 拒识逻辑可用
- SKU 管理与索引重建流程可用

从工程实现角度看，系统已经具备了“可演示、可扩展、可迭代”的基础条件。

### 6.2 当前优势

- `YOLO` 在商品目标框检测上的表现相对稳定，是当前系统中最成熟的部分
- `CLIP + FAISS` 方案具备良好的扩展性，适合持续增加 SKU
- `OCR` 能帮助区分包装颜色接近但文字不同的商品
- `Unknown` 机制能够避免部分明显错误的强行匹配
- 前后端链路已经打通，适合持续做真实业务验证

### 6.3 当前主要瓶颈

当前全库模式的主要问题不在于系统“不能识别”，而在于 **全库准确率仍明显受 SKU 参考图质量影响**。核心原因有三点：

- **域差距（Domain Gap）明显**
  - SKU 库中很多图片来自网图、标签图或电商图
  - 实际货架图存在倾斜、遮挡、反光、模糊、压缩和复杂背景
  - 两者视觉分布差异较大，导致匹配不稳定

- **SKU 参考图覆盖度不足**
  - 部分 SKU 只有少量正面图
  - 缺少轻侧拍、仰拍、俯拍、反光、弱光和货架实拍视角
  - 导致同一商品在真实场景中容易被判成 `Unknown` 或误匹配到相似包装商品

- **细粒度差异难点突出**
  - 相同品牌下的不同口味、不同容量、不同包装版本外观相近
  - 如果文字区域不清晰，仅凭颜色和布局容易混淆

因此，当前全库模式更适合表述为：

> 系统已经具备全流程识别能力，并在结构上验证了 SKU 级识别方案可行；下一阶段准确率提升的关键，不是重写架构，而是系统化提升 SKU 图库质量与覆盖度。

## 7. 准确率提升建议

这是老板汇报中最关键的部分之一。结论非常明确：**SKU 库图片越完整、越清晰、越接近真实货架视角，系统准确率就越高。**

### 7.1 优先级最高：提升 SKU 图库质量

建议将后续优化工作的第一优先级放在 SKU 参考图本身，而不是立即更换主架构。

建议每个 SKU 至少补充以下类型图片：

- 正面清晰图
- 轻微左侧角度图
- 轻微右侧角度图
- 轻微俯拍图
- 轻微仰拍图
- 带少量反光的图
- 靠近真实货架背景的图

如果资源有限，建议先从重点 SKU 开始，保证每个重点 SKU 至少有 `3-8` 张高质量多角度图；如果要提升全库表现，建议逐步把核心 SKU 扩展到 `6-12` 张高质量参考图。

### 7.2 建立统一的拍摄标准

为了避免“图库数量增加但质量没有提升”，建议对 SKU 图采集建立简单标准：

- 光线均匀，避免过曝和严重阴影
- 主体尽量完整，不被手部或其他商品遮挡
- 正面图必须清晰可读品牌名和口味名
- 角度变化要真实，避免只拍多张几乎一样的图片
- 尽量包含少量真实陈列背景，缩小网图和货架图之间的域差距

### 7.3 分批扩展，而不是一次性追求全库完美

建议采用“分层推进”的方式：

- 第一层：重点品牌 / 高频 SKU 优先优化
- 第二层：相似包装、易混淆 SKU 优先补图
- 第三层：长尾 SKU 逐步补齐

这样做的好处是：

- 能更快得到老板看得见的精度提升
- 能优先解决误识别最严重的品类
- 能在有限采集成本下获得最大收益

### 7.4 保留并继续使用 OCR 辅助

对于口味、容量、版本差异明显依赖文字区分的商品，OCR 仍然值得保留。建议策略是：

- 只在 CLIP 分数不够稳定时触发 OCR
- 重点匹配品牌名、口味名、容量信息
- 对常见品牌词建立关键词表

这能帮助系统在不大幅增加推理成本的前提下，提高细粒度区分能力。

### 7.5 继续维护 Unknown 拒识策略

在真实业务场景中，“不乱认”比“全都认”更重要。建议继续坚持：

- 分数不足时输出 `Unknown`
- 候选分差太小时输出 `Unknown`
- 批次图或异常图不满足特征一致性时拒绝入库

这会让系统在老板演示时更稳，也更符合业务可信度要求。

### 7.6 后续可选增强方向

如果后续需要继续冲击更高全库精度，可以逐步引入以下增强：

- 针对重点 SKU 建立专项评估集
- 增加真实货架标注样本做专项验证
- 对重点品类做小范围对比学习或分类微调
- 将真实货架 crop 回流到 SKU 参考库，形成持续优化闭环

但在当前阶段，最有效、投入产出比最高的措施，仍然是 **补高质量 SKU 图库**。

## 8. 风险与限制

当前项目仍存在以下现实限制：

- 全库图片来源不完全统一，质量参差不齐
- 网图和真实货架图之间仍有明显域差距
- 相似包装商品的细粒度区分仍依赖更好的参考图与文字信息
- OCR 在复杂图像上会带来一定耗时
- 当 SKU 数量继续扩大时，需要持续关注精度与速度平衡

这些限制并不否定当前方案，而是说明项目已经进入“工程优化与数据优化阶段”。

## 9. 下一阶段计划

建议后续工作按以下顺序推进：

### 第一阶段：提升图库质量

- 清洗现有 `database/sku_images`
- 删除明显不适合作参考的网图、标签图和低清图
- 为重点 SKU 补拍多角度高质量图片

### 第二阶段：建立标准评估集

- 准备一批固定的真实货架测试图
- 按“包含库内商品 / 混合场景 / 不包含库内商品”分组
- 固定评估流程，比较每次优化前后的变化

### 第三阶段：优化重点品类

- 先做饮料、泡面等高频演示品类
- 对易混淆 SKU 做定向补图和专项验证
- 形成可复用的“重点 SKU 提升模板”

### 第四阶段：准备业务化扩展

- 完善 SKU 导入与索引更新流程
- 逐步扩展更多品类与更多门店场景
- 评估部署方式和多用户测试方案

在 **Full Library Mode（全库模式）** 下，当前系统已经证明：

- 方案整体可行
- SKU 级别识别链路已打通
- 后续扩展能力具备

当前准确率提升的决定性因素，不是推翻现有架构，而是持续提升 SKU 图库的完整性、清晰度和真实场景覆盖度。换句话说，**系统框架已经搭好，下一步最值得投入的工作，是把 SKU 数据做“对、全、真”。**

如果后续按重点 SKU 优先、图库质量优先、评估闭环优先的方式持续推进，项目具备进一步提升全库识别效果并逐步走向稳定业务演示的可行路径。
